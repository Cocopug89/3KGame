## 3. Things worth flagging for whoever next touches these cards

1. **`rainingArrows.ts` has a small, real, pre-existing gap this task's tests document rather than
   paper over.** 八卦阵's `demand.open` listener fires on `kind === 'dodge'` regardless of `reasonKey`,
   so it can deem 万箭齐发's dodge demand too, not just `strike.ts`'s own. Unlike `duress.ts` (which
   checks `supplied.length > 0` before treating an answer as real), `rainingArrows.ts`'s "supplied"
   branch does not special-case an empty array — it falls into the `log.responds` branch and reads
   `supplied[0]`, which is `undefined` for a deemed `[]`. This does not throw (JS returns `undefined` for
   an out-of-bounds array index, not an error) — the practical effect is a log line with a missing card
   name, not a crash. `rainingArrows.test.ts`'s last test pins this exactly as observed, rather than
   "fixing" it silently in a test — a real fix belongs in `rainingArrows.ts` itself, and is out of scope
   for a test-only task.
2. **`frostBladeDiscardFrames`/`rockCleavingAxeHitFrames` never mutate `G` (engine-design §3) —
   `cardChoicesFor` re-derives choices from the SAME unmutated `G` on every call within one `resolve()`,
   even after a `{t:'moveCards'}` frame has been *returned* for the card just discarded.** The card is
   still physically in the hand array until the pump actually applies that frame. Both test files
   originally asserted the discarded card would already be missing from the second request's choices —
   that assumption was wrong (caught by running the tests, not by inspection) and both were corrected to
   assert both cards are still offered. This is not a bug in the game: by the time the SECOND real
   request goes out to a player, the pump has applied the first `moveCards` and the live hand really has
   shrunk — it only looks stale within a single synchronous `resolve()` call that returns multiple
   frames at once.
3. **`harvest.ts`'s "advance with nothing chosen" branch requires `G.revealed` to still be non-empty.**
   The first version of that test used a `G` with `revealed: []` (the fixture default) and got `[]` back
   instead of the expected continuation frame — not a bug in `harvest.ts`, just a fixture that didn't
   match the precondition the branch actually needs (`idx >= order.length || G.revealed.length === 0` is
   checked before the "already asked" logic). Fixed by setting `G.revealed` to a non-empty array in that
   one test.
4. **`eightTrigramsTrigger` (`content/effects/eightTrigrams.ts`) never sets an explicit `priority`,
   unlike every other equipment trigger in 3.6, which pins `PRIORITY_EQUIPMENT` explicitly per
   CONTINUE.md's own note ("equipment must set this explicitly — missing ⇒ 200, the skill band").**
   Not fixed here (out of scope for a test-only task) — flagged for whoever next touches this file. The
   test file asserts `optional: false` and the actual `when`/`effect` behavior, not a `priority` this
   task did not want to assume.

5. **`duel.test.ts` needed a post-hoc fix — not a bug in this task, but worth recording since the real
   `run-tests.bat` run is what caught it, not this session's own scratch-build verification.** Between
   this task writing its tests and the real Windows test run, a concurrent lane landed a change to
   `duel.ts` (documented in its own inline comment): the damage frame duel emits now carries
   `card: ctx.cards[0]` — the original 决斗 card — so that 裸衣-shaped skills can tell "this damage came
   from a 杀 or 决斗" apart from card-less AoE damage. Two of this file's assertions (the "current fails
   to supply" and "backfires onto the turn player" cases) were written against the pre-change shape and
   failed on the real run; both were updated to expect the `card` field. **This is the exact "a shared
   file moved under you mid-session" scenario CONTINUE.md warns about** — `duel.ts` is task 3.4's file,
   not this task's, and the fix was to make the test agree with the (correct, deliberate) current
   behavior, not to touch `duel.ts` itself.
6. **The rest of the real `run-tests.bat` run's failures are NOT this task's** — recorded here only so a
   future session doesn't re-diagnose them. At the time this task's tests were verified: `npm run build`
   failed on real (pre-existing) TypeScript errors in `src/content/effects/lordProxy.ts` and
   `src/content/skills/fanjian.ts`, neither touched by this task; and `test/bgio/{moves,game}.test.ts`,
   `test/bgio/stages.test.ts` and `test/bgio/endgame.test.ts` (none written or owned by this task) failed
   because the skill registry is no longer empty (4.2–4.4 landed real skills) and those pre-existing
   tests' fixtures/drivers/stage-list assertions haven't caught up — e.g. `driveOneRequest` doesn't know
   how to answer a `confirmSkill` prompt, and `stages.test.ts` expects a shorter stage list than
   `THREE_KINGDOMS_STAGE_MOVES` now has. All 20 of this task's own files (18 unaffected, `duel.test.ts`
   fixed as above) pass; these others belong to whichever lane is running 4.2–4.4.
