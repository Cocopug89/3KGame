## 2. The two real bugs found, and how they were verified

Both of these are genuine, currently-would-fail-on-first-deploy bugs inherited from task 0.3 / the
phase-0-1-audit fix ŌĆö not hypothetical. Found by actually running each platform's exact committed build
command in a clean scratch copy, not by reading the config and guessing.

### Bug 1: `client/netlify.toml` built `client/` in isolation, which can't resolve `@3k/shared`

The old file's `[build] command = "npm run build"` combined with a dashboard "Base directory: client"
setting means the build runs as `cd client && npm run build` (`tsc -b && vite build`). But `client/src/`
imports `@3k/shared` throughout (`game/cardIndex.ts`, `lobby/*.ts`, most of `components/table/`) ŌĆö added
across Phases 2ŌĆō6, after task 0.3 wrote the original `netlify.toml`. `@3k/shared`'s `package.json` points
`main`/`types` at `./dist/shared/src/...`, which only exists after `shared`'s own `tsc` runs. A build
scoped to `client/` never builds `shared` first.

**Verified failing:** ran `cd client && npm run build` from a fresh scratch install ŌåÆ 20 `TS2307: Cannot
find module '@3k/shared'` errors, exit code 1.
**Verified fixed:** ran `npm run build -w shared && npm run build -w client` from the scratch root ŌåÆ
clean build, `client/dist/{index.html,assets/}` produced (`index-*.js` ~407KB, `index-*.css` ~9.5KB).

### Bug 2: `server/render.yaml`'s buildCommand had the identical bug, server-side

`buildCommand: npm ci && npm run build -w server` ŌĆö same missing `shared` step. `server/src` imports
`@3k/shared` in `bgio/game.ts`, `content/skillSource.ts`, `engine/{cardIndex,selection,setup}.ts`,
`server.ts`.

**Verified failing:** ran `npm run build -w server` alone (after a fresh install) ŌåÆ same class of
`TS2307` errors (11 of them) plus knock-on `any`-typed-parameter errors from the missing types.
**Verified fixed:** ran `npm run build -w shared && npm run build -w server` ŌåÆ clean build,
`server/dist/server.js` produced. Then actually **booted it** (`PORT=8099 node server/dist/server.js`)
and hit it: printed the expected `­¤Ä« Three Kingdoms Kill Server running on port 8099` / lobby-ready log
lines, and `curl -X POST http://localhost:8099/rooms?numPlayers=4` returned a real room
(`{"roomCode":"HLTVK","matchID":"...","numPlayers":4,"seats":[...],...}`). This is the same code path a
live Render deploy runs (`npm start -w server` ŌåÆ `node dist/server.js`), just not on Render's actual
infrastructure.

Both fixes are the same shape: **`shared` must build before either `client` or `server`**, and the old
configs never encoded that dependency. This is now spelled out in both config files' comments and in
`DEPLOY.md`'s troubleshooting table, so a future edit to either build command doesn't reintroduce it
silently.

## 3. Why the files moved to the repo root

- `netlify.toml`: Netlify reads config from the "Base directory" (default: repo root) only ŌĆö a file
  living in `client/` requires the dashboard's Base directory to be set to `client` for Netlify to find
  it at all, which is a manual, undocumented-in-code dependency. Root placement needs zero dashboard
  config beyond leaving Base directory blank.
- `render.yaml`: Render's Blueprint feature (`New +` ŌåÆ `Blueprint`) only auto-detects `render.yaml` at
  the repo root. It was previously in `server/`, where it would never have been picked up by a Blueprint
  deploy ŌĆö only the fully-manual dashboard path (Option B in `DEPLOY.md`) would have worked, silently
  making the committed file dead weight for anyone who tried the faster route.

## 4. Server requirements this config assumes ŌĆö read before changing service type

- **`type: web` (Render), not a static site or background worker, is load-bearing.** boardgame.io's
  server needs a persistent, long-running Node process for two reasons: (1) socket.io ŌĆö every game move
  rides one open WebSocket connection per player, not request/response; (2) **all match and room-code
  state is in-memory** (`server/src/lobby/roomCodes.ts`, boardgame.io's default match store) ŌĆö there is
  no database anywhere in this stack. A serverless/edge function model would lose all of it between
  invocations. This also means: **a Render redeploy or a free-tier 15-minute idle spin-down silently
  ends every in-progress match.** No warning to players, no persistence to recover from. Acceptable for a
  friends-only playtest (7.2); worth flagging if this ever needs to be "always on."
- **`PORT` must not be hardcoded anywhere in config.** `server/src/server.ts` already reads
  `process.env.PORT` with a `3000` fallback for local dev only. Render injects `PORT`; setting it as an
  env var on the dashboard is the classic "no open ports detected" failure mode (this was bug #2 in the
  original phase-0-1-audit, and the new `render.yaml` comments call it out again so it doesn't recur).
- **`CLIENT_ORIGIN` gates both the WebSocket handshake and the REST lobby routes** (`POST /rooms`,
  `GET /rooms/:code`) ŌĆö they share one Koa router and one `origins` array passed to boardgame.io's
  `Server()` in `server.ts`. One env var, both surfaces. If it's unset, `server.ts` logs `ŌÜĀ’ĖÅ No allowed
  origins configured` on boot ŌĆö that's the one-line diagnostic to grep Render's logs for.
- **`VITE_SERVER_URL` must be `https://`, never `wss://` or bare `http://`** in production.
  `client/src/lobby/serverUrl.ts`'s `toHttpUrl()` defaults a bare `host:port` to `http://`, which the
  browser blocks as mixed content on an `https://` Netlify site. This was bug #3 in the original
  phase-0-1-audit (`[env.production]` wasn't even a real netlify.toml section, so a wrong value there was
  silently ignored) ŌĆö the new `netlify.toml` deliberately does *not* commit a value, forcing it through
  the dashboard (or the documented `[context.production.environment]` block) where it's visible.

## 5. What's still manual ŌĆö this sandbox cannot do these

No credentials exist in this environment for GitHub, Netlify, or Render. Concretely, a human needs to:

1. **Push this repo to GitHub** (or GitLab/Bitbucket). `docs/version-control.md` only ever ran `git init`
   locally (`git-setup.bat` + `autosnapshot.bat`) ŌĆö there is no remote configured. Both platforms' "connect
   repo" flow needs a hosted repo to point at. This is the actual first blocker on a from-scratch deploy,
   ahead of anything in `netlify.toml`/`render.yaml`.
2. **Create the Netlify site** (Option in `DEPLOY.md`'s "Client: Netlify" section) and confirm the
   auto-detected build settings from the root `netlify.toml` match (base directory blank, command
   `npm run build -w shared && npm run build -w client`, publish `client/dist`).
3. **Create the Render service**, either via Blueprint (finds `render.yaml` automatically) or the manual
   dashboard path ŌĆö both documented in `DEPLOY.md`.
4. **Resolve the CORS chicken-and-egg**: deploy Render first (its URL needs nothing external), set
   `VITE_SERVER_URL` on Netlify to that URL and deploy the client, then set `CLIENT_ORIGIN` on Render to
   the real Netlify URL (triggers Render's auto-redeploy). `DEPLOY.md` has this as its own callout section
   since it's easy to deploy both once and have neither know about the other.
5. **Run the actual end-to-end test** in `DEPLOY.md`'s "Testing the end-to-end connection" section ŌĆö
   `?phase0` counter smoke test first, then the real two-tab lobby ŌåÆ general-selection ŌåÆ play-a-card flow.
   Nothing in this sandbox could exercise this because it requires two real browser sessions hitting live
   URLs.

## 6. Assumptions made, for the record

- **Node 20** is pinned via `NODE_VERSION` in both configs. Not derived from an `.nvmrc` or `engines`
  field (neither exists in the repo ŌĆö root `package.json` sets none) ŌĆö chosen as a current Node LTS
  compatible with the toolchain in use (Vite 5, TS 5.2, ESM throughout, `boardgame.io` 0.50). If a real
  deploy hits a Node-version-specific failure, this is the first thing to revisit; it was not verified
  against Render's or Netlify's actual default Node image, only against whatever Node the sandbox itself
  runs.
- **`plan: free` on Render** ŌĆö matches the original task-0.3 intent ("acceptable for testing," per the
  pre-7.1 `DEPLOY.md`). The in-memory-state-loss-on-spin-down consequence is now documented explicitly
  (┬¦4) rather than left implicit.
- **The SPA redirect rule** (`/* ŌåÆ /index.html`, status 200) in `netlify.toml` is currently unnecessary ŌĆö
  `client/src/main.tsx` routes entirely by query string (`?table`, `?gallery`, `?phase0`), not by path ŌĆö
  but costs nothing and protects against a future path-based route or a deep-link refresh 404ing on a
  static host. Kept from the original config rather than removed.
- **No health-check path configured for Render.** Render's default behavior (checking that the assigned
  port is listening) is sufficient here; the app has no dedicated `/healthz` endpoint to point at, and
  adding one was judged out of scope for "first real deploy."
