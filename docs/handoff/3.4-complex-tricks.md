### 2b. `server/src/engine/frames.ts` — the reveal primitive + `revealed` zone

**THE reveal-primitive design call** (CONTINUE.md / build-breakdown.md's old 3.3b row asked 3.4 to make
it): a 4th primitive, `{t:'reveal', count}`, resolved in `pump.ts` where the `rng` lives — **not** a
`count`-carrying variant of `moveCards`. Full reasoning is in the frame's own doc comment; short version:
`moveCards` always names the exact ids it's moving (that's what lets an effect return it without touching
`G`), and a reveal can't know those ids in advance without the rng `CardEffect.resolve()` isn't allowed to
touch (engine-design §3).

```ts
export type Zone =
  | { z: 'hand' | 'equip' | 'judgementZone'; player: PlayerId }
  | { z: 'discard' | 'drawPile' | 'revealed' };
```

```ts
  | { t: 'reveal'; count: number }
```

If `Frame`'s union or `Zone`'s union is missing the `revealed`/`reveal` arm, harvest.ts won't type-check.

### 2c. `server/src/engine/pump.ts` — three additions

**(i) `'revealed'` zone in `takeFromZone`/`putInZone`** (the two functions that back `{t:'moveCards'}`):
takeFromZone splices an id out of `G.revealed`; putInZone pushes onto it (nothing in 3.4 actually routes
*into* `revealed` this way — the `{t:'reveal'}` primitive populates it directly — but a future effect
reusing the pool has somewhere correct to land).

**(ii) the `'reveal'` case itself**, mirroring `drawTop()`'s reshuffle-on-empty behaviour but stopping
early (not throwing) if both piles run dry:

```ts
    case 'reveal': {
      const revealed: CardId[] = [];
      for (let i = 0; i < frame.count; i++) {
        if (G.drawPile.length === 0) {
          if (G.discardPile.length === 0) break;
          G.drawPile = shuffleDeck(G.discardPile, rng);
          G.discardPile = [];
        }
        revealed.push(G.drawPile.shift() as CardId);
      }
      G.revealed = revealed;
      return;
    }
```

**(iii) U1 — `legalTargets` filled in at `'request'` pop time**, not by whoever pushed the frame (a card
played earlier in the same batch, e.g. 无中生有's draw, must already have happened):

```ts
      G.pending =
        frame.req.kind === 'act'
          ? { ...frame.req, legalTargets: legalTargetsForHand(G, frame.req.playerId) }
          : frame.req;
```

plus the import `import { legalTargetsForHand } from './legalTargets.js';`. `playerView` (bgio/game.ts,
§2f below) already only forwards the full `pending` object to the player it names — U1 needed **no**
anti-cheat change, it rides the existing filter.

## 3. Decisions inside 3.4, so nobody relitigates them

1. **The reveal primitive is `{t:'reveal', count}`, not a `count`-carrying `moveCards`.** See §2b/§2c.
   `harvest.ts` never touches `G.drawPile`/`rng` directly — it returns the primitive and reads
   `G.revealed` back on its next `resume`.
2. **五谷丰登 self-wraps exactly ONE window**, around the reveal only — `pump.ts`'s `'play'` case does
   **not** wrap a `nullify:'custom'` card in anything. `harvest.ts`'s `resolve()` pushes its own
   `nullifyWindowFrame` on the *first* call (guarding the reveal itself — one 无懈可击 cancels the whole
   card, nobody sees any of the revealed cards), then re-enters with `revealed:true` and does the actual
   reveal + picks, which are **not** nullifiable past that point (matches design §2's "一张锦囊牌对一名角色
   产生的效果" not applying once the shared reveal already happened for everyone).
3. **决斗/南蛮入侵/万箭齐发/借刀杀人 all reuse the EXISTING `{t:'demand', kind:'strike'|'dodge'}` channel** —
   no new demand kind was needed. `kind` is just an effectKey string fed through `queries.cardsAs`, so
   视为杀/视为闪 skills (武圣, 龙胆, etc., Batch A/B) answer these demands for free, same as they do for a
   plain 杀/闪.
4. **决斗's exchange is a ping-pong `resume` loop**, not recursion — `current`/`other` swap on every
   successful supply, and the loop only ever terminates by someone failing to supply (one hit, no
   cumulative damage). F1 (3.2) is what makes a losing 决斗 backfiring onto the turn player safe.
5. **南蛮入侵/万箭齐发/桃园结义 use `nullify:'per_target'`** (judgement-nullification-design §2.2): `pump.ts`
   wraps each target in its OWN window and calls `resolve()` once per target with a single-element
   `targets` array, so these files only ever think about one target at a time.
6. **乐不思蜀/闪电 both set `nullify:'none'`, which is NOT the trick default** — the real 无懈可击 window for
   a delayed trick opens at the start of the victim's judge phase (phases.ts, §2f above), not at play time.
   Leaving `nullify` unset here would double-window the card (once wrongly at play time, once correctly at
   judge time). This is judgement-nullification-design §3's documented timing trap, called out explicitly
   in both files' headers.
7. **闪电's `source` on the damage frame is `null`** — 闪电 has no killer, so Phase 5's kill reward/penalty
   never fires for a thunder-strike death. `DamageKind` already had `'thunder'` (frames.ts, predates 3.4).
8. **借刀杀人's second target really gets struck** — `targets[0]` playing "yes" re-enters the engine via a
   real `{t:'play', effectKey:'strike'}` against `targets[1]`, not a bespoke damage frame. Dodge, 龙胆-style
   视为闪, everything strike.ts already handles applies.

## 4. Known simplifications and gaps (documented, not silently accepted)

1. **AoE target-count is not enforced server-side.** `TargetSpec.max: 'all_others'`/`'all'` can't express
   "must be exactly every other/all living player" independent of client behavior — a hand-crafted move
   could under-target a 南蛮入侵/万箭齐发/桃园结义. A future client that always offers the full living-player
   set is what makes these AoEs in practice. (barbarianInvasion.ts / rainingArrows.ts / peachGarden.ts
   headers.)
2. **借刀杀人's pairwise range constraint is checked at `resolve()` time, not target-selection time** —
   `TargetSpec.predicate` only ever sees one candidate against `self`, never against the other selected
   target, so "targets[1] must be in targets[0]'s attack range" can't be expressed there. An out-of-range
   or now-weapon-less pairing simply fizzles at resolve. "Once per action phase" for 借刀杀人 is **not**
   enforced at all (duress.ts header).
3. **五谷丰登's `{z:'revealed'}` chooseCard slot has no client renderer yet** — same shape as 3.3's original
   `chooseCard` gap before 6.4b closed it. The server-side flow is complete; a table that plays 五谷丰登
   today stalls on whichever player is asked to pick. Flagged for whichever session next opens
   `PromptPanel`/`ChoicePanel`.
4. **This session's bash sandbox could not complete a build/test run.** `npm ci && npm run build` in a
   fresh `/tmp` scratch (rsync'd from the mount) failed on `client/src/game/log.ts` with a syntax error that
   turned out to be a torn read (the mount's bash-visible copy was truncated mid-token; the real file, read
   via the file tools, was complete and correct). A repo-wide scan afterward found **~20 more files** in the
   same torn state on this session's bash mount — `server/src/engine/{pump,frames,legalTargets,cardChoice,
   phases,setup,state}.ts`, `server/src/content/{effectRegistry,skillRegistry,equipmentQueryRegistry}.ts`,
   all 3.4's own effect files, and five pre-existing server test files unrelated to this task
   (`chooseCard.test.ts`, `game.test.ts`, `moves.test.ts`, `fixtures.ts`, `pump.test.ts`) — all confirmed
   complete and correct via the `Read` tool, all truncated mid-token in bash. This matches CONTINUE.md's own
   documented gotcha ("an agent's sandbox reads this folder over a mount that returns stale, truncated
   content... neither npm nor git can be trusted from there") almost exactly, at a larger scale than
   previously logged. **Two of the torn files (`locales/en.json`, `locales/zh.json`, and separately
   `client/src/game/log.ts`) were fixed in place** (correct content rewritten to the real mount path,
