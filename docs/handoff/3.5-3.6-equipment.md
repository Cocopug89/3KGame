### 2b. `server/src/engine/frames.ts` — one new Frame variant

**This is the one piece of 3.6 that is NOT confined to `content/`.** 八卦阵 needs to *deem* a demand
answered without any card leaving anyone's hand, and no `CardEffect`/`SkillTrigger` may mutate `G`
directly (engine-design §3) — so this needed a genuine new primitive, not a workaround. It is
pre-authorized by `docs/skill-trigger-design.md` §12.2 for exactly this card; it is the third instance
of the `{t:'setDamage'}`/`{t:'retrial'}` pattern (a listener-mutation channel for a window already open),
not a new kind of thing.

Inserted right after `| { t: 'demandClose' }` and before the `// ── primitives` comment:

```ts
  /** Task 3.6's one addition to the demand protocol: the sanctioned mutation
   * channel for a `demand.open` listener that DEEMS the demand answered
   * without a card leaving anyone's hand — 八卦阵's judgement becoming a 闪 is
   * the only Standard user. A CardEffect/SkillTrigger may never mutate G
   * directly (engine-design §3); this is the {t:'setDamage'}/{t:'retrial'}
   * pattern's third instance. `cards: []` is a valid DEEMED answer — see
   * DemandInfo.supplied's own doc comment on why `[]` and `null` are not the
```

## 3. The decisions inside 3.5/3.6, so nobody relitigates them

1. **Equip-first, handlers-first-within-that: `equip.ts` was written before any of the 11 handler
   files, and `effectRegistry.ts` (the one contended file) was edited LAST, after every handler file
   already existed and compiled in isolation.** Rationale: the registry is the single highest-collision
   file in this task's own footprint (four lanes editing it concurrently), so minimizing the time
   between "read it" and "write it" minimizes the clobber window. Writing every new file first, then
   doing the registry edit as the very last surgical step, means the read-edit-write round trip is one
   Edit call instead of eleven separate ones spread across the session.
2. **`moveCards`'s equip-slot replace-on-equip needed no change.** `putInZone`'s `'equip'` case in
   `pump.ts` (task 3.1/3.2 vintage — see §2's zone plumbing) already discards whatever was in the slot
   before writing the new card in. Verified rather than assumed: read the function, confirmed the
   behavior, wrote no code for it. This is the one piece of "3.5's equipment zone" that turned out to
   already exist.
3. **All 8 handlers with a `trigger` half are `locked`-shaped where they touch a query, per
   `LOCKED_ONLY_QUERIES`** (engine/queries.ts's four un-askable folds: `strikeLimit`, `drawCount`,
   `demandCount`, `targetLimit`, plus `cardsAs`/`targetable`/`ignoresDistance`/`distanceModifier` are
   OR/AND-folded and don't require locked, but the four chained ones do). Equipment is definitionally
   锁定技 in Standard, so this was never a design choice, just an assertion to get right.
4. **Ordering hazard between three `damage.before` listeners, solved with explicit priorities, not seat
   order.** 青釭剑 (100) must set `ignoreArmour` before 仁王盾 (105) reads it; 仁王盾's prevention must be
   decided before 麒麟弓 (110) discards a horse (no point discarding a horse for a strike that's about to
   be fully blocked, and the Standard card text implies the horse-discard *replaces* the block, not
   stacks with it — this ordering makes them mutually exclusive on any single strike, which matches the
   real card). `PRIORITY_EQUIPMENT = 100` is the band; the +5/+10 offsets are these three handlers only.
5. **Three multi-step "ask and come back" patterns, one shared shape:** frost_blade (discard up to 2 of
   the TARGET's cards), rock_cleaving_axe (discard up to 2 of the OWNER's OWN cards, then hit anyway),
   gender_swords (the TARGET picks their own card to hand over). All three reuse
   `cardChoicesFor`/`resolveSlot`/the `chooseCard` request from 3.3's slot protocol
   (`engine/cardChoice.ts`) rather than inventing a new request kind. gender_swords is the interesting
   case: `chooseCard`'s request shape allows `playerId === target` (the answerer and the zone-owner are
   the same player) — nothing in `bgio/game.ts`'s move enforces they differ — so "the target chooses
   which of their own cards to give up" needed zero new engine surface, just noticing that self-choice
   was already legal.
6. **八卦阵 needed a real new primitive, and everything else didn't.** This is worth being explicit about
   because "add a primitive" is the kind of thing that should be rare and justified: every other handler
   fits inside triggers + queries + the existing frame vocabulary. 八卦阵 is the one Standard equipment
   piece whose whole effect is "the demand.open fan-out silently deems this answered" — which needed a
   channel to write `G.demand.supplied` from inside a trigger's returned frames, since a trigger's
   `effect()` may not touch `G` directly. `{t:'demandSupply'}` is that channel — see §2b/§2c above.
   `docs/skill-trigger-design.md` §12.2 pre-authorizes exactly this ("a deemed card (八卦阵) can make an
   otherwise un-answerable demand answerable"), so this is filling in a documented gap, not scope creep.

## 4. Documented simplifications — flagged for a future session if a richer choice primitive ever exists

None of these are wrong per the Standard card text's common interpretation, but each picked the simpler
of two legal readings because the engine currently has no "offer player X a choice of N named options and
get one back" primitive that isn't `chooseCard`'s card-slot shape:

1. **绿龙偃月刀 (green_dragon_blade)'s bonus strike does not restrict the second strike to the same
   target as the first.** The turn flag counts "how many extra strikes are available," and the
   `strikeLimit` query just adds that count — it does not thread *which* target triggered it. A stricter
   reading (bonus strike must retarget the same player) would need the flag to carry a target id and the
   query to somehow constrain `act`'s target list, which the query fold's `targetLimit`/`strikeLimit`
   split doesn't currently support in one hook.
2. **麒麟弓 (unicorn_bow) picks a horse deterministically when the target has both a +1 and a -1 horse**
   (checks `plusHorse` first). The real card lets the attacker choose which one. A real choice would be
   another `chooseCard`-shaped request; deferred because it's a genuinely rare board state (both horse
   slots filled on the same target) and the deterministic pick is still a legal, non-cheating outcome.
3. **雌雄双股剑 (gender_swords) never offers the "let the attacker draw a card instead" branch when the
   target does have a card to give up** — the target is always asked to choose one of their own cards.
   The real card gives the *target* that choice (give a card, or let the striker draw). This would need
   a yes/no fork before the chooseCard request, which is mechanically easy (an extra `confirmSkill`-shaped
   step) but was left out to keep the handler to one request round-trip; flagged here rather than silently
   dropped.

None of these need an engine change to fix — they need either an extra request round-trip (gender_swords)
or a small ctx field threaded through an existing flag (green_dragon_blade, unicorn_bow). Left as
follow-ups rather than blocking 3.5/3.6's own definition of done.
