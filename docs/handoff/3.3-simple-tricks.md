### 2c. `server/src/bgio/game.ts` — one import + one stage

Import (next to the other `../engine/` imports):

```ts
import { resolveSlot, type CardSlot } from '../engine/cardChoice.js';
```

Stage, inside `turn.stages`, between `demandCard` and `discard`:

```ts
      // "Point at one of that player's cards" — 过河拆桥/顺手牵羊 (task 3.3,
      // docs/judgement-nullification-design.md §5). Answered by the player who
      // PLAYED the card, not the one losing it.
      //
      // The answer is a SLOT, never a card id: the target's hand is hidden, and
      // ids leak suit and rank (playerView, engine-design §6), so a hand card is
      // addressed by its position in the server's own `player.hand` array. The
      // server maps it back to a real id here — engine/cardChoice.ts's
      // resolveSlot, which re-validates against live state, so a hand-crafted
      // slot naming an out-of-bounds index or unworn equipment is INVALID_MOVE
      // rather than a crash or a leak.
      chooseCard: {
        moves: {
          chooseCard: ({ G, ctx, random, events, playerID }, slot: CardSlot) => {
            if (!G.pending || G.pending.kind !== 'chooseCard' || G.pending.playerId !== playerID) {
              return INVALID_MOVE;
            }
            const target = G.pending.target as PlayerId;
            const resolved = resolveSlot(G, target, slot);
            // No "decline" branch, unlike respondDodge/respondPeach: once the
            // card is played and a target is chosen, taking a card is mandatory.
            if (!resolved) return INVALID_MOVE;

            applyToResumeFrame(G, { chosen: resolved.cardId, chosenZone: resolved.zone });
            G.pending = null;
            const rng = makeRng(random as BgioRandomLike);
            pump(G, rng);
            syncBgio(G, ctx, events);
            return undefined;
          },
        },
      },
```

## 3. The three decisions inside 3.3, so nobody relitigates them

1. **A hand card is addressed by INDEX, never by id.** Card ids leak suit and rank by construction
   (`peach_3h`), and `playerView` bans masking (engine-design §6), so the request enumerates opaque hand
   positions plus the *public* zones by id. **The index is the server's own `player.hand` order** — never
   re-sort a hand per client, or the same index means two different cards on the two ends of the wire.
   `resolveSlot` maps it back and re-validates against live state, so a hand-crafted slot is
   `INVALID_MOVE`, not a leak. **4.4's 大乔/陆逊 reuse this — do not write a second one.**
2. **过河拆桥 and 顺手牵羊 are one effect** (`takeOneCardEffect()`), differing only in range
   (`inRange: 'distance_1'`) and destination zone (discard vs. the thief's hand). Both re-check the victim
   **at pick time** — still alive? still holds a card? — because targeting proved that when the card was
   *played*, and the entire 无懈可击 chain runs in between. "Target has a card" is a `predicate` on the
   TargetSpec, so `validateTargets` enforces it for free.
3. **All three leave `nullify` unset.** The `type: 'trick'` default is `'once'` (effectTypes.ts), which
   pump reads — the cards get their 无懈可击 window without knowing it exists. The tests pin the default
   rather than the value.

## 4. What 3.3 did NOT do: 五谷丰登, and why it's 3.4's

Not laziness, and not just "the `revealed` zone isn't built." The reveal step must take N cards off the
draw pile, and **a short draw pile has to reshuffle the discard pile in — which needs the `rng`**.
`CardEffect.resolve()` has no rng and may not mutate `G` (engine-design §3). So its setup frame *cannot*
be a plain `{t:'moveCards', from:{z:'drawPile'}}` with ids read out of `G.drawPile` at resolve time.

3.4 has to **decide**: a 4th primitive (`{t:'reveal', count}`, resolved in `pump.ts` where the rng lives),
or a `count`-carrying variant of `moveCards`. It's a design call, which is why it belongs with the other
complex tricks rather than being smuggled in here.
