# Build Breakdown — Phases, Tasks & Model Assignments

Companion to [three-kingdoms-plan.md](three-kingdoms-plan.md) §7. Each task is tagged with the cheapest model that can deliver it reliably.

**This table is the single source of truth for project progress.** Every session — human or agent — should update the Status column before signing off. See [`CONTINUE.md`](../CONTINUE.md) at the repo root for the fast-orientation protocol.

Status legend: ✅ done · ◀ NEXT · ⬜ pending

## Model strategy (token efficiency)

| Model | Use for | Why it's efficient |
|---|---|---|
| **Haiku 4.5** | Mechanical transcription: JSON data files, locale strings, boilerplate, config, repetitive unit tests from specs | Cheapest per token; these tasks are spec→output with no design judgment |
| **Sonnet 5** | Default builder: engine code, React UI, most skill handlers, debugging, integration | Strong coding at mid price; handles 80% of implementation without escalation |
| **Opus 4.8 / Fable 5** | Design docs, complex-logic review, security audit | Short, high-leverage outputs only. Never use for bulk codegen — a 1-page design from Opus that prevents a Sonnet rewrite is the best token trade in the project |

**Rules of thumb:** default to Sonnet; drop to Haiku when the task is pure transcription of an existing spec; escalate to Opus/Fable only when the output is a *decision* (architecture, timing rules, security), then hand the decision back to Sonnet/Haiku to implement. Expected mix: ~30% Haiku, ~60% Sonnet, ~10% Opus/Fable.
